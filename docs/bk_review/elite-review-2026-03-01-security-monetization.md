## 致命的な問題（CRITICAL）

### SEC-C1: RSO認証ユーザーへのパスワードログイン拒否が未実装

**該当ファイル**: `src/app/api/auth/callback/riot/route.ts` L100-110

**前回指摘**: CRITICAL-3（一部改善のまま残存）

**問題**: `app_metadata: { auth_method: 'rso' }` は設定済みだが、**ログインフロー側でこのメタデータを検証していない**。攻撃者が合成メール `rso_${puuid}@lolcoach.ai` でSupabaseのパスワードリセットフローを実行した場合、RSOユーザーのアカウントを乗っ取れる可能性がある。

**Agent S**: puuidはRiot APIで半公開情報だ。合成メールパターンが `rso_` + puuid と予測可能である以上、**パスワードリセットフローを通じたアカウント乗っ取りは現実的な脅威**だ。

**攻撃シナリオ**:
1. 攻撃者がターゲットのpuuidを取得（Riot API等で公開情報に近い）
2. `rso_${puuid}@lolcoach.ai` でSupabaseのパスワードリセットAPIを呼ぶ
3. ただしメールは実在しないため配達されない → **現時点では攻撃不成立**
4. **しかし**、将来的にキャッチオールメール設定や、Supabaseのメール設定変更で攻撃面が広がるリスク

**改善案（工数: 2時間）**:
1. ログインServer ActionまたはSupabase auth hookで `app_metadata.auth_method === 'rso'` のユーザーのパスワードログインを拒否
2. パスワードリセットAPIでも同様にRSOユーザーを除外
3. 合成メールにランダムソルトを含める: `rso_${puuid}_${randomBytes(4).toString('hex')}@lolcoach.ai`

**深刻度**: CRITICAL（アカウント乗っ取りの潜在リスク）
**緊急度**: MEDIUM（現時点では合成メールが配達されないため即時の脅威は低い）

### SEC-C2: CSP `script-src 'unsafe-inline'` が本番環境で有効

**該当ファイル**: `next.config.ts` L14

```typescript
`script-src 'self' 'unsafe-inline'${isDev ? " 'unsafe-eval'" : ''} https://js.stripe.com ${sentryOrigin}`,
```

**Agent S**: `unsafe-eval` を開発環境限定にしたのは正しい。だが**`unsafe-inline`が本番でも有効なのは見落としだ**。これはXSSの主要な緩和策であるCSPの効果を大幅に弱体化させる。攻撃者がHTML injection経由で `<script>alert(document.cookie)</script>` を注入した場合、CSPが防がない。

**Agent M**: セキュリティ問題としては深刻だが、マネタイゼーションへの直接的影響は？

**Agent S**: ユーザーのセッションCookieが盗まれれば、Premium課金ユーザーのアカウントを乗っ取って解約させることも可能。間接的に収益に影響する。

**改善案（工数: 4時間）**:
1. Next.jsのnonce-based CSPに移行（`experimental.sri` または `metadata` API）
2. Stripe.jsは既に外部ドメインから読み込みなので `'unsafe-inline'` は不要
3. インラインスタイルはTailwind CSSのクラスベースなので `style-src` からも `'unsafe-inline'` を除外可能

**深刻度**: CRITICAL
**緊急度**: HIGH（DOMPurifyが防御層として機能しているため即時の脆弱性ではないが、防御の多層化として必須）

### SEC-H2: フレームアップロードのサイズ制限とbodySizeLimitの不整合

**該当ファイル**: `src/lib/validation.ts` L29-32, `next.config.ts` L55

**Agent S**: Zodスキーマでは `framesArraySchema = z.array(base64FrameSchema).max(30)` で各フレーム最大2MB × 30枚 = 理論上60MBのリクエストが可能。しかし `bodySizeLimit` は5MB。

```typescript
// validation.ts
const base64FrameSchema = z.string().max(2 * 1024 * 1024); // 2MB per frame
export const framesArraySchema = z.array(base64FrameSchema).max(30);

// next.config.ts
bodySizeLimit: 5 * 1024 * 1024, // 5MB
```

**実際の影響**: クライアント側でフレームを個別に送信している場合は問題ない。だがZodスキーマが「30フレーム一括送信OK」と定義している以上、クライアントの実装変更で5MB制限に引っかかる可能性がある。

**改善案（工数: 30分）**:
1. `framesArraySchema` の最大枚数をbodySizeLimitと整合させる: `max(2)` で1リクエスト2フレーム（4MB）
2. または実際の送信パターンに合わせてZodスキーマを修正

**深刻度**: HIGH（DoS経路になり得る）
**緊急度**: LOW（現時点では5MB制限が防御として機能）

---

### SEC-H3: ログインページにレート制限なし

**Agent S**: Supabase自体にはGoTrue側のレート制限があるが、アプリケーション層での制限がない。`supabase/config.toml` でメール送信は2回/時に制限されているが、パスワードログイン試行自体は無制限。

**改善案（工数: 2時間）**:
1. Middlewareにログイン試行回数の制限を追加（IP単位で10回/分）
2. Upstash Ratelimit（@upstash/ratelimit）などのエッジ対応ライブラリを使用
3. 5回失敗後にCAPTCHA表示

**深刻度**: HIGH（ブルートフォース攻撃の経路）
**緊急度**: MEDIUM

**改善案**:
1. **Free分析を月2回に変更**（週1→月2）。「毎週使えるほどではないが、価値を体感できる」バランス
2. **Premiumを週10回に減らし、月額¥680に**。CVRを上げる価格設定
3. または **Free分析をゼロにし、「分析プレビュー」のみ表示**: グレード/バッジまでは無料で見えるが、詳細フィードバックはブラー

### MON-C2: ゲスト分析成功後のコンバージョンCTAが弱すぎる

**該当ファイル**: `src/app/analyze/page.tsx`, `src/app/actions/guestAnalysis.ts`

**Agent M**: ゲスト分析は「3回無料」で体験させる仕組みだが、**分析結果表示後のアップセル設計がほぼない**。クレジット不足時にメッセージを出すだけでは遅い。

現状のユーザーフロー:
```
ゲスト → 分析実行 → 結果表示（全内容表示） → ...そして何もない → 離脱
```

あるべきフロー:
```
ゲスト → 分析実行 → 結果のサマリー表示 →
  「詳細を見るには無料登録」→ 登録 →
  「試合履歴で継続追跡するにはPremium」→ 課金
```

**改善案（工数: 8時間）**:
1. **ゲスト分析結果の部分ブラー**: grade/badge/laneVerdictは表示するが、keyFeedbackの後半50%をブラー + 「無料登録で全文を見る」
