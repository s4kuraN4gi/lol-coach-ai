## 総合評価: 要改善（複数のHIGH/CRITICALが存在）

---

## 1. 認証・認可

### [CRITICAL] SEC-001: Riot Sign-On (RSO) のパスワード生成に公開鍵を使用

**ファイル**: `src/app/api/auth/callback/riot/route.ts` 90行目

```typescript
const dummyPassword = Buffer.from(puuid + process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY).toString('base64');
```

**問題**: RSO経由のユーザーに対して、`puuid`（Riot API経由で公開取得可能）と`NEXT_PUBLIC_SUPABASE_ANON_KEY`（クライアントサイドに公開されている）をBase64結合してパスワードとして使用している。これは暗号学的に全く安全ではない。攻撃者は任意のRSOユーザーのpuuidさえ知っていれば、そのSupabaseアカウントに`signInWithPassword`でログインできる。

**影響**: 全RSOユーザーのアカウント乗っ取りが可能。

**修正方法**: Service Role Clientの`auth.admin.createUser`/`auth.admin.generateLink`を使用し、パスワードレスな方式でユーザー作成・セッション発行を行う。または、`crypto.randomUUID()`等で推測不可能なパスワードを生成し、DB等に安全に保管する。

---

### [HIGH] SEC-002: OAuth Callback のオープンリダイレクト脆弱性

**ファイル**: `src/app/auth/callback/route.ts` 8行目

```typescript
const next = searchParams.get('next') ?? '/dashboard'
// ...
return NextResponse.redirect(`${origin}${next}`)
```

**問題**: `next`パラメータをバリデーションせずにリダイレクト先として使用。`?next=//evil.com` でユーザーを攻撃者のサイトにリダイレクト可能。

**修正方法**:
```typescript
const next = searchParams.get('next') ?? '/dashboard';
const safeNext = next.startsWith('/') && !next.startsWith('//') ? next : '/dashboard';
```

---

### [HIGH] SEC-003: RSO認証フローにCSRF対策（stateパラメータ）が未実装

**ファイル**: `src/app/api/auth/riot/route.ts`

**問題**: Riot OAuth認可リクエストに`state`パラメータが含まれていない。CSRF攻撃で攻撃者のRiotアカウントを被害者のSupabaseアカウントに紐づけることが可能。

**修正方法**: ランダムな`state`値を生成し、Cookieに保存してからRiotに送信。コールバックで一致を検証する。

---

### [HIGH] SEC-005: RSO Callback でエラー詳細がURLに漏洩

**ファイル**: `src/app/api/auth/callback/riot/route.ts` 207行目

**問題**: サーバーサイドのエラーメッセージ（スタックトレース含む可能性）がURLパラメータ経由でクライアントに漏洩。

**修正方法**: ユーザーにはエラーコードのみ返す。詳細はサーバーサイドのログにのみ記録。

---

### [MEDIUM] SEC-004: AuthProviderで`getSession()`を使用（非推奨）

**ファイル**: `src/app/Providers/AuthProvider.tsx` 23行目

**問題**: Supabase公式ではクライアントサイドの`getSession()`はローカルストレージからトークンを読み取るだけで、サーバーとの検証を行わない。改ざんされたJWTをそのまま信頼する可能性。

---

## 2. APIセキュリティ

### [HIGH] SEC-006: Riot API `tagLine` パラメータ未エンコード

**ファイル**: `src/app/actions/riot.ts` 47行目

**問題**: `gameName`は`encodeURIComponent`でエンコードされているが、`tagLine`はそのまま展開。パストラバーサル/SSRFの入口になりうる。

---

### [HIGH] SEC-007: Server Actionレベルでのレートリミット未実装

**問題**: Gemini APIへのリクエストにアプリケーションレベルでのレートリミットが存在しない。TOCTOU競合により、10個の並行リクエストで1回分のカウントで10回のGemini API呼び出しが可能。

---

### [HIGH] SEC-008: `getAnalysisJobStatus`でユーザー所有権チェックが不完全（IDOR）

**ファイル**: `src/app/actions/analysis.ts` 275-289行目

**問題**: `.eq("user_id", user.id)` がクエリに含まれていない。RLSが未設定の場合、任意のユーザーの分析結果を閲覧可能。

---

### [HIGH] SEC-011: `upgradeToPremium()` がモック実装として残存

**ファイル**: `src/app/actions/analysis.ts` 149-175行目

**問題**: 認証済みユーザーなら誰でもServer Actionを直接呼び出してプレミアム状態にできる。`is_premium: true`, `analysis_credits: 999`が設定される。

**影響**: 全ユーザーが無料でプレミアム機能を利用可能。直接的な収益損失。

---

### [HIGH] SEC-015: IP ベースのゲストクレジットが容易に回避可能

**ファイル**: `src/app/actions/guestCredits.ts`

**問題**:
1. X-Forwarded-Forスプーフィング可能（`CF-Connecting-IP`を優先すべき）
2. インメモリフォールバック -- サーバー再起動でクレジットリセット
3. エラー時に`credits: 3, canUse: true`を返す（Fail-Open設計）

---

### [MEDIUM] SEC-009/010/013/016/017/019/020/021

- 100MB Body Sizeリミット（DoSベクター）
- Stripe Secret Keyプレフィックスの変数代入
- `dangerouslySetInnerHTML`でサニタイズなし（4箇所）
- Service Role Clientのスコープが広すぎる
- `.env`ファイルがDockerビルドコンテキストに含まれる可能性
- Stripe初期化のダミーキーフォールバック
- Webhook署名検証の空文字フォールバック
- TOCTOU競合によるクレジット多重消費

# Part 2: マネタイゼーション戦略レビュー

**審査員**: マネタイゼーション戦略家

---

## 総合評価: D+ (収益化の骨格はあるが、致命的な構造問題が複数存在)

---

## 1. 料金設計 [CRITICAL]

### 現状の料金体系

| プラン | 価格 | 分析回数/週 | セグメント数 |
|--------|------|-------------|-------------|
| ゲスト | 無料 | 3日に1回(最大3回貯蓄) | 2個 |
| 無料会員 | 無料 | 3回/週 | 2個 |
| Premium | 980円/月 | 20回/週 | 4個 |
| Extra | 2,980円/月 | 50回/週 | 5個 |

### 致命的問題

**無料プランが手厚すぎる。** カジュアルLoLプレイヤーが週に3回以上の動画分析をする動機がほぼ存在しない。つまり**ほぼ全てのユーザーにとって無料プランで用が足りてしまう。**

**Premium と Free の差分が弱い。** 週20回の分析が必要なユーザーはほぼいない。ビルドアドバイスはop.ggやu.ggで無料で手に入る。

**Extra(2,980円/月)の価値が不透明。** 月額3倍の価格差に対して、追加価値がダメージ計算機能1つだけ。

### 修正案
- 無料プランの分析回数を **週1回** に制限
- ミクロ分析をPremium以上限定に
- 分析結果の「深さ」で差別化
- 年額プラン追加(20%オフ)

---

## 2. 決済インフラ [HIGH]

### 良い点
- Stripe Checkout + Billing Portal が正しく実装
- proration（日割り計算）実装済み

### 問題点
1. **payment_method_types を `['card']` にハードコード** -- 日本のLoLプレイヤー(10代後半~20代)はクレジットカード非保有率が高い。コンビニ決済が必須
2. **NEXT_PUBLIC_STRIPE_PRICE_ID をフロントから送信** -- 悪意あるユーザーが任意のPrice IDを送信可能
3. **Stripe初期化のフォールバック値 `sk_test_dummy`** が危険

---

## 3. ゲスト分析が寛容すぎる [CRITICAL]

- サーバー再起動でクレジットリセット（デプロイ毎に無限クレジット）
- VPNでIP変更すれば無制限
- **ミクロ分析（最もコストの高い機能）がゲスト/無料で使える** -- APIコストだけが増えてマネタイズできない構造

---

## 4. チャーンリスク [HIGH]

1. AIアドバイスが一般論レベルなら価値を感じない
2. 動画録画→アップロード→分析のフローが重い
3. 成果の可視化がない（何ランク上がったか追跡不可）
4. 週20回の上限を使い切るユーザーはほぼいない

---

## 5. コスト分析 [CRITICAL]

### マージン計算

| プラン | 月額 | APIコスト/月(推定) | 粗利率 |
|--------|------|-------------------|--------|
| 無料 | 0円 | 6円 | -100% (赤字) |
| Premium | 980円 | 15円 | 98.5% |
| Extra | 2,980円 | 70円 | 97.7% |

**結論: APIコスト自体は問題ない（Gemini Flashは安い）。問題は無料ユーザーが有料化しないこと。** 無料100人に対して有料1人だと、100人分のAPIコスト(600円/月)を1人の980円でカバーする計算になり、インフラコスト含めると赤字。

### [CRITICAL] アイコン変更による認証 - 最大の離脱ポイント

### [MEDIUM] ウィジェット9つで情報過負荷

### [MEDIUM] DDragonバージョンのハードコード散在

## 5. レースコンディション: クレジット消費 [HIGH]

Read-Modify-Write パターンで実装。同時リクエストで1クレジットしか消費されない。

**修正方法**: Supabase RPCでアトミックなデクリメントを実装:
```sql
CREATE OR REPLACE FUNCTION decrement_credits(user_id UUID)
RETURNS INT AS $$
UPDATE profiles SET analysis_credits = GREATEST(analysis_credits - 1, 0)
WHERE id = user_id AND analysis_credits > 0
RETURNING analysis_credits;
$$ LANGUAGE sql;
```

---

## 6. 大量のコード重複 [HIGH]

Premium/Free判定 + クレジット消費パターンが `analyzeVideo`, `analyzeMatchQuick`, `analyzeMatch` の3箇所で完全に重複。`analyzeMatch` では旧式の `daily_analysis_count` がまだ使われている。
